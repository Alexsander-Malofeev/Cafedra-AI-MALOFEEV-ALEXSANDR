# Отчёт по проекту

## 1. Постановка задачи

Проект посвящён классификации пользовательских обращений в службу поддержки. Пользователь пишет текстовое сообщение, а система должна определить категорию обращения.

Это BASIC-версия темы 5.1 «Классификация обращений/тикетов». В отличие от более сложной версии, в проекте не определяется приоритет обращения и не используется несколько тегов. Основная цель — сделать понятный end-to-end сервис: данные, модель, API, тесты и базовая наблюдаемость.

## 2. Пользователь сервиса

Пользователем может быть сотрудник службы поддержки или helpdesk-система. Сервис помогает быстрее распределять обращения по типам проблем.

## 3. Данные

Используется синтетический датасет `data/tickets.csv`. Он создаётся скриптом:

```bash
python -m src.generate_dataset
```

В датасете две колонки:

- `text` — текст обращения;
- `category` — правильная категория.

Категории:

- `payment_issue`;
- `refund_request`;
- `delivery_issue`;
- `account_access`;
- `technical_bug`;
- `general_question`.

Датасет не содержит персональных данных.

## 4. Предобработка текста

Перед обучением текст приводится к более простому виду:

- нижний регистр;
- замена `ё` на `е`;
- удаление лишней пунктуации;
- удаление лишних пробелов.

## 5. Модель

Используется простая ML-модель для классификации текста:

- TF-IDF по словам;
- TF-IDF по символам;
- Logistic Regression.

Символьные признаки помогают частично учитывать простые опечатки, например `вирнуть` вместо `вернуть`.

## 6. Оценка качества

После обучения модель проверяется на тестовой части датасета. В `artifacts/metrics.json` сохраняются основные метрики, а в `artifacts/classification_report.txt` — текстовый отчёт по категориям.

Команда обучения:

```bash
python -m src.train
```

## 7. Сервис

Сервис реализован на FastAPI. Основной endpoint:

```text
POST /predict
```

Пример входа:

```json
{
  "text": "Не могу войти в аккаунт"
}
```

Пример выхода:

```json
{
  "category": "account_access",
  "predicted_category": "account_access",
  "confidence": 0.93,
  "needs_clarification": false,
  "message": "Категория определена."
}
```

Если текст слишком короткий или модель не уверена, сервис возвращает категорию `needs_clarification` и просит пользователя уточнить обращение.

## 8. Наблюдаемость

В проекте есть базовые элементы наблюдаемости:

- логирование предсказаний;
- `GET /health` для проверки состояния сервиса;
- `GET /metrics` для базовых Prometheus-метрик.

## 9. Риски и ограничения

Главные ограничения:

- данные синтетические;
- реальные обращения могут быть сложнее;
- короткие сообщения часто нельзя корректно классифицировать без контекста;
- обработка опечаток реализована только на базовом уровне.

## 10. Что можно улучшить дальше

- Добавить больше синтетических и реальных обезличенных примеров.
- Добавить отдельное определение приоритета обращения.

## 11. Разведочный анализ данных и эксперименты

Для дополнительной проверки данных и сравнения моделей был добавлен ноутбук:

`notebooks/main.ipynb`

В нём выполняется разведочный анализ данных: проверяется размер датасета, структура колонок, наличие пропусков и дубликатов, распределение категорий, длина текстов и примеры обращений по каждому классу.

Также в ноутбуке сравниваются несколько моделей: простой baseline `DummyClassifier`, базовая модель `CountVectorizer + LogisticRegression`, улучшенная модель `TF-IDF word n-grams + LogisticRegression` и финальная модель `TF-IDF word + char n-grams + LogisticRegression`.

Результаты экспериментов сохраняются в файл `artifacts/experiment_results.csv`, а краткая информация о лучшей модели сохраняется в `artifacts/experiment_summary.json`.

Сравнение baseline-модели и улучшенных моделей нужно для проверки того, что финальная модель действительно лучше простого подхода. В качестве основных метрик используются `accuracy`, `f1_macro` и `f1_weighted`. Для многоклассовой классификации особенно важна метрика `f1_macro`, так как она учитывает качество по всем категориям, а не только по самым частым классам.

## 12. Обоснование выбора финальной модели

В качестве финальной модели используется подход `TF-IDF word + char n-grams + Logistic Regression`.

Эта модель выбрана потому, что TF-IDF по словам хорошо подходит для коротких текстовых обращений, n-граммы по словам помогают учитывать устойчивые фразы, а символьные n-граммы частично помогают при опечатках и разных формах слов. Logistic Regression выбрана как простая, быстрая и воспроизводимая модель, которую удобно сохранять и использовать внутри API-сервиса.

Такой вариант хорошо подходит для учебного end-to-end проекта, потому что он закрывает полный цикл: подготовка данных, обучение модели, сравнение с baseline, сохранение артефактов и использование модели через FastAPI.

## 13. Артефакты экспериментов

После запуска ноутбука `notebooks/main.ipynb` создаются дополнительные файлы:

`artifacts/experiment_results.csv` — таблица с результатами сравнения моделей.

`artifacts/experiment_summary.json` — краткая информация о лучшей модели и её метриках.

Эти файлы дополняют основные артефакты обучения: `artifacts/metrics.json`, `artifacts/classification_report.txt` и `artifacts/ticket_classifier.joblib`.

## 14. Итог

Проект реализует полный цикл AI Engineering: генерация и подготовка данных, разведочный анализ, обучение модели, сравнение baseline и улучшенных моделей, сохранение артефактов, FastAPI-сервис для инференса, базовая наблюдаемость, тесты, Dockerfile и документация.

Таким образом, проект соответствует формату end-to-end ML-сервиса и демонстрирует не только работу API, но и процесс выбора модели на основе экспериментов.
